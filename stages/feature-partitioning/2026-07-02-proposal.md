# feature-partitioning 提案（2026-07-02）

## 背景

intent（`intent/INTENT.md`、`intent/NON_GOALS.md`、`intent/DESIGN_PRINCIPLES.md`）で、LLMGPはLLMSRの作り直しであり、開発は段階的に進める（まずLLMなしのGPフレームワークを完成させ、一通りの機能を確認してからLLMを導入する）方針が承認された。

この方針を踏まえ、今回のfeature-partitioningでは、土台となる「LLMなしGPフレームワーク」を5つのfeatureに分割して開発順序を確定する。intentに記載されている6つの追加機能（複数ケース評価、複数LLM比較、複数評価地点、収束履歴の可視化、既存モデルとのベンチマーク比較、LLM併用）は、土台の完成後に必要になった時点で個別にfeature-partitioningし直す方針とし、今回のfeature_orderには含めない。

## 分割結果

| feature名 | 内容 |
|---|---|
| `term-representation` | 項（遺伝子）のデータ構造。命令列（load・mul・div・exp・log・べき乗などの演算子）と、内部に自由な係数（b, c, dなど）を持てる形式を定義する |
| `term-variation-operators` | 項の初期生成（seeding）、交叉（1点交叉）、変異（命令の一部をランダムに変更） |
| `equation-fitting` | 複数の項を線形に組み合わせて方程式を作り、DE（差分進化法）で全係数（項内部の係数＋外側の重み）を一括最適化する。評価関数にはL1罰則（不要な項の重みを0に近づける）を含める |
| `evolution-loop-and-selection` | 世代交代のループ本体。子の生成（term-variation-operatorsを使用）→方程式の組み立てと評価（equation-fittingを使用）→エリート戦略による選択、を繰り返す |
| `term-importance` | 最適化後の項の重みを正規化し、項の重要度を算出する |

## 依存関係の主張と理由

| feature名 | 依存先 | 依存の種類 | 理由 |
|---|---|---|---|
| `term-representation` | なし | - | 他のすべての基礎となるデータ構造のため、依存を持たない |
| `term-variation-operators` | `term-representation` | hard | 項のデータ構造（命令列の形式）を直接操作する処理であり、その構造なしには実装できないため |
| `equation-fitting` | `term-representation` | hard | 項を数値的に評価する（データに当てはめる）ためには、項のデータ構造と、そこから数式を組み立てる方法が定義されている必要があるため |
| `evolution-loop-and-selection` | `term-variation-operators` | hard | 子（次世代の項）を生成するために、交叉・変異の操作が必要なため |
| `evolution-loop-and-selection` | `equation-fitting` | hard | 生成した項の集団を方程式として評価するために、係数最適化とスコア計算の仕組みが必要なため |
| `term-importance` | `equation-fitting` | hard | 重要度は、係数最適化の結果得られる項の重み（β）を正規化して算出するため、equation-fittingの出力なしには計算できない |

## 開発順序の導出

依存関係から、`term-representation`を最初に置く（依存を持たないため）。次に、`term-representation`にのみ依存する`term-variation-operators`と`equation-fitting`を置く（依存関係上はどちらが先でもよいが、交叉・変異という「項の生成」を先に扱う方が、その後の係数最適化の入力となる項を用意する流れとして自然なため、`term-variation-operators`を先に置く）。両方に依存する`evolution-loop-and-selection`はその後に置く。最後に、`equation-fitting`の出力を使う`term-importance`を置く（`evolution-loop-and-selection`が完成し、実際に世代を重ねた項に対して重要度を確認できる状態の方が、動作確認がしやすいため）。

1. `term-representation`
2. `term-variation-operators`
3. `equation-fitting`
4. `evolution-loop-and-selection`
5. `term-importance`

## 今回含めない機能（将来の候補）

intentに記載済みの次の6機能は、土台の完成後、必要になった時点で個別にfeature-partitioningを行う。

1. 複数ケース間の汎化性能評価
2. 複数LLMの比較
3. 収束履歴の可視化
4. 複数の評価地点
5. 既存モデルとのベンチマーク比較
